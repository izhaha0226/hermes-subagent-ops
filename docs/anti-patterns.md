# 자주 나오는 반패턴

이 문서는 Hermes/Prime 중심 분신 운영에서 자주 발생하는 실패 패턴을 정리한다.

## 1. 말뿐인 분신

증상:
- frontend clone, backend clone 이름은 있는데 실제 worktree path가 없음
- 결국 하나의 작업공간을 여러 역할이 번갈아 사용함

문제:
- 상태 추적이 안 된다
- 누가 무엇을 바꿨는지 흐려진다
- 병렬 작업이 아니라 충돌 예약 상태가 된다

대응:
- 분신명보다 먼저 실제 path/branch/worktree를 만든다

## 2. 분신의 자가 완료 선언

증상:
- qa clone이 "완료"라고 말함
- deploy clone이 "배포 완료"라고 최종 보고함

문제:
- 사용자 기준 최종 판단권이 여러 개가 된다
- 검증 책임이 흐려진다

대응:
- 분신은 `Prime 검토대기`까지만 상태를 올린다
- 완료 상태는 Prime만 선언한다

## 3. 테스트 통과 = 서비스 성공 착각

증상:
- 로컬 테스트 통과 후 바로 완료 처리
- 배포 로그만 보고 실사용 흐름은 확인하지 않음

문제:
- 실제 장애를 놓친다
- 완료 보고의 신뢰도가 급격히 떨어진다

대응:
- 테스트/빌드/배포/실서비스 확인을 분리한다
- 최소 1개 실제 사용자 흐름 확인 전에는 완료 표현을 금지한다

## 4. 자율루프의 활동량 착시

증상:
- 실행 횟수, 수정 파일 수, 커밋 수만 많이 남음
- 전후 변화나 성과 기록은 없음

문제:
- 많이 움직였지만 무엇이 좋아졌는지 설명할 수 없다
- 다음 루프가 학습되지 않는다

대응:
- 모든 루프에 `전후 변화`와 `다음 액션`을 강제한다
- 실패는 gotcha, 성공은 best practice로 축적한다

## 5. 역할명과 실제 최신 상태 불일치

증상:
- frontend clone이 최신일 것이라 가정했는데 실제 수정은 debug clone에 있음
- deploy clone 기준선이 오래되어 반영 충돌 발생

문제:
- 작업 시작 전에 이미 잘못된 전제를 깔게 된다
- cherry-pick, merge, 배포 검증이 모두 꼬인다

대응:
- 시작 전 각 분신의 branch, HEAD, dirty 상태를 먼저 점검한다
- 역할명보다 현재 상태를 우선한다
