# 공용 분신술 운영 스킬

## 개요
분신술의 핵심은 "에이전트를 많이 부르는 것"이 아니라, 같은 판단 기준을 가진 실행 축을 분리해 속도와 검증력을 같이 얻는 것이다.

공식:
- Dev-Prime = 최종 판단/보고/검증 책임자
- 분신 = 역할이 분리된 실행 공간
- 보고는 항상 Dev-Prime만 사용자에게 한다

즉,
- 사용자 → Hermes(Dev-Prime)
- Hermes → 분신(frontend/backend/debug/qa/deploy)
- 분신 → Hermes에게만 결과 보고
- Hermes → 최종 보고

## 언제 쓰나
- 사용자가 "분신술", "클론 체계", "병렬로 굴리자", "24/7 자율실행"을 원할 때
- 하나의 레포를 프론트/백엔드/디버그/QA/배포처럼 동시에 굴려야 할 때
- 서브에이전트보다 실제 git worktree 기반 물리 분리가 더 안전할 때
- 배포/검증/수정이 서로 꼬이지 않게 분리해야 할 때
- 특정 프로젝트가 아니라 여러 프로젝트에 공통으로 재사용할 운영 모델이 필요할 때

## 핵심 원칙
1. 분신은 다른 성격의 에이전트가 아니다
- 같은 코어
- 같은 보고 기준
- 같은 진실성 규칙
- 역할만 다름

2. 말뿐인 분신 금지
- 대시보드에 보이면 실제 worktree/브랜치/상태가 있어야 한다
- "frontend clone" 이라면 진짜 경로와 브랜치가 있어야 한다

3. 기준은 root agent(Prime) 하나
- 분신이 사용자에게 직접 완료 보고하지 않는다
- 완료 판단은 항상 root agent(Prime)가 검증 후 내린다
- 어떤 분신도 스스로 완료를 확정하지 않는다

4. 분신 역할명보다 실제 최신 구현 상태를 먼저 본다
- frontend 분신이 항상 프론트 최신이라는 보장은 없다
- backend 분신이 항상 백엔드 최신이라는 보장도 없다
- 작업 전 반드시 각 분신의 브랜치/dirty 상태/최근 커밋을 본다

5. 24/7 자율실행은 "많이 돌리는 것"보다 "전후 비교 학습"이 중요하다
- 조치
- 결과 측정
- 원인 해석
- 다음 액션
이 루프가 있어야 한다

## 추천 표준 분신 구조
기준 저장소: <repo-root>

1. frontend 분신
- UI, 페이지, 컴포넌트, 프론트 빌드 검증

2. backend 분신
- API, DB, 스케줄러, 비즈니스 로직

3. debug 분신
- 재현, 로그 분석, 루트코즈 조사

4. qa 분신
- 회귀 확인, 핵심 흐름 점검, 완료 전 검증

5. deploy 분신
- 배포선 반영, healthcheck, 실서비스 확인

권장 브랜치명:
- worktree/frontend-clone
- worktree/backend-clone
- worktree/debug-clone
- worktree/qa-clone
- worktree/deploy-clone

## Prime 승인 게이트
모든 분신 작업은 아래 게이트를 통과해야만 완료로 인정된다.

### 게이트 1. 분신 실행 완료
- 분신은 자기 역할 범위의 실행만 끝낸다
- 이 단계에서는 `완료`라는 표현을 쓰지 않는다
- 허용 표현:
  - `수정 완료, Prime 검증 대기`
  - `재현 완료, Prime 판단 대기`
  - `배포 수행 완료, Prime 실서비스 확인 대기`

### 게이트 2. Prime 증거 검토
Prime가 반드시 직접 확인한다.
- 변경 파일
- 테스트/빌드 결과
- 배포 결과
- 실서비스 확인 결과
- 남은 리스크

### 게이트 3. Prime 완료 선언
Prime만 아래 상태 중 하나를 선언할 수 있다.
- 완료
- 조건부 완료
- 보류
- 실패

어떤 분신도 자기 결과만으로 `완료`를 선언하지 않는다

## 완료 상태 머신
- 대기
- 실행중
- Prime 검토대기
- 완료
- 조건부 완료
- 보류
- 실패

규칙:
- 분신은 `Prime 검토대기`까지만 올릴 수 있다
- `완료/조건부 완료/보류/실패` 전환은 Prime만 가능하다
- 사용자에게 노출되는 최종 상태도 Prime 선언만 인정한다

## 시작 전 체크리스트
실제 적용 전에 최소한 아래는 먼저 확인한다.

- Prime가 누구인지 명확한가
- 분신별 실제 worktree path가 있는가
- 분신별 브랜치명이 정해져 있는가
- 각 분신의 현재 dirty/clean 상태를 확인했는가
- 사용자에게 노출될 보고 채널은 Prime 하나로 고정됐는가
- 완료 판단 기준(테스트, 빌드, 배포, 실서비스 확인)이 정의됐는가

## 24/7 자율실행 운영법
권장 루프:
1. 상태 확인
2. 이번 조각 선택
3. 해당 분신에서 수정
4. qa 분신 또는 최소 검증 수행
5. deploy/main 반영
6. 실서비스 확인
7. 전후 성과 또는 시스템 변화 기록
8. 다음 액션 결정

주의:
- 변경량이 많다고 진전이 큰 것이 아니다
- 결과 로그가 없으면 자율운영이 아니라 작업 흔적 수집에 가깝다
- 실패는 다음 루프의 입력값으로 반드시 남긴다

## 보고 규약
분신 보고는 항상 다음 5개 축을 유지하는 것을 권장한다.

1. 맡은 역할
2. 수행 내용
3. 증거
4. 미검증 영역
5. Prime에게 필요한 다음 판단

Prime 최종 보고는 항상 다음 순서를 권장한다.

1. 문제
2. 원인
3. 바꾼 것
4. 검증 결과
5. 남은 리스크
6. 다음 액션

## 한 줄 요약
분신술은 에이전트를 많이 만드는 기술이 아니라, Dev-Prime 하나를 중심으로 실제 worktree 분신들을 정직하게 운영해 속도, 검증, 배포, 학습을 동시에 잡는 공용 운영 기술이다.
